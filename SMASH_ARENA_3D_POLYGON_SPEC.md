# SMASH ARENA 改修仕様書

作成日: 2026-06-15
対象: `standalone.html`
確認元: https://github.com/CatwalkTK/test/blob/claude/charming-gauss-4gn7z8/standalone.html

## 1. 目的

現行の「SMASH ARENA」は、Canvas 2D の単体 HTML として、プラットフォーム対戦アクションに必要な基本機能をすでに備えている。一方で、音源、画面演出、キャラクター表現はプロトタイプ感が強く、対戦ゲームとしての重厚感が不足している。

本仕様では、既存のゲーム性を維持しながら、以下を強化する。

- 音源を軽い合成音から、打撃感・低音・空間感のあるサウンドへ変更する。
- 画面表現を平面的な Canvas 2D から、奥行きと光源のある 2.5D 表現へ引き上げる。
- キャラクターを丸形状の手描きパーツから、低ポリゴン 3D 風モデルに近い表現へ変更する。
- 物理、操作、CPU、ストック制など現行のゲームルールは大きく変えない。

## 2. 現行ゲームの確認結果

### 2.1 現行機能

- キャラクター選択画面
  - 8体のファイター
  - 1P vs CPU / 2P 対戦
  - CPU 難易度切り替え
  - ストック数切り替え
  - ミュート切り替え
- バトル
  - 2D 横視点の対戦アクション
  - ステージ中央のメイン足場と、上部のすり抜け足場
  - ダメージ%によるノックバック増加
  - ストック制
  - ガード、シールド耐久、シールドブレイク
  - 弱、ダッシュ、上強、スマッシュ、上スマッシュ、空中攻撃、下空中メテオ、飛び道具
  - CPU 操作
- 演出
  - ヒットストップ
  - 画面揺れ
  - KO フラッシュ
  - パーティクル
  - WebAudio による簡易 BGM / SE

### 2.2 確認した画面

- 選択画面: `output/playwright/select.png`
- 戦闘画面: `output/playwright/battle.png`

### 2.3 現行の弱点

- 音源
  - WebAudio の単純な oscillator / noise が中心で、音色が軽い。
  - BGM は短い単音シーケンスで、ベース、ドラム、コード、展開が足りない。
  - 打撃音に低域、トランジェント、残響、レイヤーが不足している。
  - KO、スマッシュ、シールドブレイクなどの重要イベントに固有の迫力が足りない。

- 画面表現
  - 背景は要素数が多いが、ステージとキャラクターが平面的で主役の密度が弱い。
  - ライティング、影、被写界深度、ポストエフェクトがない。
  - 足場が単純な角丸長方形で、バトルアリーナとしての存在感が弱い。
  - ヒット時の衝撃表現はあるが、火花、残像、軌跡、衝撃波の層が少ない。

- キャラクター表現
  - `paintCharacter()` による Canvas 手描きで、丸胴体、丸頭、線状の腕脚が中心。
  - キャラごとのシルエット差はあるが、立体感と素材感が弱い。
  - 武器や装飾は記号的で、重量級、スピード型、メカ型などの個性が画面上で弱い。
  - アニメーションは手足の簡易補間が中心で、攻撃前後の溜め、反動、着地重量が不足している。

## 3. 改修方針

### 3.1 推奨表現

ゲームプレイは現行の 2D 横視点を維持し、見た目だけを 2.5D 化する。

- 操作、当たり判定、ステージ座標は 2D のまま維持する。
- 表示は Three.js などの WebGL レイヤーで、低ポリゴン風 3D キャラクターと 3D ステージを描画する。
- カメラは横視点の Orthographic Camera を基本にする。
- Z 軸は見た目の奥行きとエフェクト分離に使い、ゲーム判定には使わない。

この方針により、既存のゲームバランスや CPU を活かしながら、画面だけを大きくリッチにできる。

### 3.2 技術スタック

推奨:

- Vite + TypeScript
- Three.js
- Howler.js または WebAudio API の自前ミキサー
- 3D アセット形式: glTF / GLB
- 画像テクスチャ: WebP / PNG
- 音源: Ogg Vorbis + MP3 フォールバック

単体 HTML 継続の場合:

- Three.js を CDN から読み込む構成でも初期移行は可能。
- ただし、キャラクター、音源、エフェクト、ステージの規模が増えるため、最終的にはモジュール分割した Vite 構成を推奨する。

## 4. ゲーム仕様

### 4.1 維持するゲームルール

- 画面サイズ基準: 960 x 540
- 固定タイムステップ: 60 FPS
- 重力、落下速度、摩擦、空中抵抗の考え方は維持する。
- ダメージ%が高いほど吹っ飛びが強くなる。
- ストック制を維持する。
- 1P vs CPU / 2P 対戦を維持する。
- CPU 難易度は easy / normal / hard を維持する。
- 操作体系は現行を維持する。

### 4.2 改修対象外

初回改修では以下を変更しない。

- オンライン対戦
- キャラクター追加
- 入力リマップ UI
- ストーリーモード
- ランキング
- 複雑な 3D 移動

## 5. 画面表現仕様

### 5.1 全体アートディレクション

方向性:

- 低ポリゴン 3D アリーナ
- 明るいファンタジー寄りだが、質感は軽すぎない
- プラットフォーム対戦アクションとしての読みやすさを優先
- 遠景は雰囲気、近景は判定と足場の視認性を優先

避けること:

- 背景だけが派手で、キャラクターが埋もれる画面
- 過度なブルームで判定が見えない状態
- キャラクターの色差だけに依存する識別
- 足場の接地ラインが曖昧になる装飾

### 5.2 カメラ

- 横視点の Orthographic Camera を標準とする。
- 戦闘中のカメラはステージ全体を常に見せる。
- ヒット時のみ短時間のカメラシェイクを入れる。
- KO 直前、スマッシュヒット時は 4 から 8 フレーム程度の軽いズームまたは FOV 演出を許可する。
- 操作中にカメラが大きく回転してはいけない。

### 5.3 ステージ

現行:

- メイン足場: 横長の角丸矩形
- ソフト足場: 3つ

改修後:

- メイン足場は浮遊島または石造アリーナとして 3D 化する。
- 上面は明確な接地ラインを持つ。
- 側面は低ポリゴン岩、金属、草などの素材差を持たせる。
- ソフト足場は薄い 3D 板またはエネルギーブリッジとして表現する。
- 接地可能な境界は、常に明るい縁取りで見えるようにする。

必要なステージ要素:

- 上面メッシュ
- 側面メッシュ
- 接地ライン発光
- キャラクター影を受ける透明シャドウ面
- 背景と分離するための輪郭ライト

### 5.4 背景

現行の空、山、雲、浮遊島のテーマは維持する。

強化内容:

- 背景の主素材は Image2 などの画像生成で作成したリアル寄りの山岳・空・雲画像を使用する。
- 初期生成アセットとして `assets/backgrounds/mountain-sky-realistic-v1.png` を使用候補にする。
- 空は 3D スカイドームまたはレイヤー化した背景にする。
- 雲は低ポリゴンの半透明モデル、または複数レイヤーのビルボードにする。
- 遠景の浮遊島はゆっくり動く 3D オブジェクトにする。
- 太陽光方向を固定し、ステージとキャラクターに同じ光源が当たるようにする。

実装方針:

- まずは Canvas の `drawSkyBackground()` で生成背景画像を全面描画する。
- 画像ロード完了前、または読み込み失敗時は現行の Canvas 背景をフォールバックとして描画する。
- 戦闘キャラと足場が埋もれないよう、背景の下部 25 から 35% に薄い暗色グラデーションを重ねる。
- 画面手前のステージ、キャラクター、ヒットエフェクトは背景画像より高コントラストにする。
- 生成画像内にプラットフォーム、人物、文字、UI、建物、武器を入れない。

生成プロンプト方針:

- 16:9 のワイド背景。
- リアルな山岳、青空、自然な雲、遠景の空気遠近を指定する。
- 下部 30% はステージと HUD を重ねやすいよう、情報量を抑える。
- `no text, no watermark, no people, no UI, no platform` を明示する。

### 5.5 ポストエフェクト

最低限:

- Bloom: 必殺技、シールド、KO、UI の一部に限定
- Vignette: ごく弱く
- Color grading: 青一色に寄せず、空、草、岩、キャラ色が分離する調整
- Hit flash: ヒット時に 2 から 4 フレームの白または色付きフラッシュ

任意:

- Motion trail
- Shockwave
- Chromatic aberration はスマッシュや KO の瞬間だけに限定

## 6. キャラクター表現仕様

### 6.1 基本方針

既存 8 キャラの性能と名前は維持し、見た目を低ポリゴン 3D 風に再設計する。

モデル仕様:

- 頭、胴体、腕、脚、武器を分離パーツ化する。
- ボーンありの GLB が理想だが、初期版はパーツ階層アニメーションでもよい。
- シルエットでキャラ識別できることを必須にする。
- 色だけでなく、体格、武器、頭部、背面装飾で差をつける。

### 6.2 キャラ別の見た目方針

ノヴァ:

- バランス型の剣士。
- 中量級の鎧、赤いスカーフ、片手剣。
- 攻撃軌跡は白から赤のスラッシュ。

ジン:

- スピード型。
- 細身の忍者/ランナー体型。
- バイザー、爪、細い発光ライン。
- 残像を短く出し、移動速度を視覚化する。

ガル:

- 重量級。
- 岩または獣人風の大きな上半身。
- 拳が大きく、着地時に小さな地面割れエフェクト。
- モーションは遅いが、溜めと反動を大きくする。

ピコ:

- ジャンプ型。
- 小柄で羽つき。
- 空中時に羽ばたき、軽い風エフェクト。
- 地上より空中でシルエットが広がる。

ミオ:

- テクニカル型。
- フード、マント、杖。
- 魔法粒子、円形ルーン、紫と金のアクセント。
- 攻撃判定は見やすい発光円弧で表示。

ロク:

- 高火力型。
- 炎の髪、拳、火のオーラ。
- 攻撃時に赤橙の火花と熱歪み風エフェクト。
- 防御面の脆さは見た目には過剰に表現しない。

リン:

- コンボ型。
- 二刀流、軽装、ピンク系アクセント。
- 連続攻撃で短い複数スラッシュを重ねる。
- 体格は細く、ポーズは前傾。

ボルト:

- リーチ型。
- メカ頭部、腕部キャノン、アンテナ。
- 電撃エフェクト、青緑の発光。
- 飛び道具は球ではなく、圧縮された電撃弾にする。

### 6.3 アニメーション

必須アニメーション:

- Idle
- Run
- Jump start
- Air loop
- Land
- Jab
- Dash attack
- Tilt / Up tilt
- Smash charge
- Smash release
- Aerial neutral
- Aerial up
- Aerial down
- Special projectile
- Guard
- Shield break
- Hit stun
- KO launch
- Victory pose

アニメーション要件:

- 攻撃前の溜め、ヒット瞬間、戻りを明確に分ける。
- スマッシュ攻撃はリリース前に 6 から 10 フレームの視覚的な圧を出す。
- ヒット時は攻撃側と被弾側の両方に短い反動を入れる。
- 着地時は重量級ほど沈み込みを大きくする。

## 7. エフェクト仕様

### 7.1 ヒットエフェクト

通常ヒット:

- 火花 8 から 16 個
- 短い白フラッシュ
- 小さな衝撃円
- ヒット音と同期

スマッシュヒット:

- 大きなスラッシュ軌跡
- 放射状の火花
- 画面揺れ
- 低音の衝撃音
- 6 から 10 フレームのヒットストップ

メテオ:

- 下方向の稲妻または矢印状エフェクト
- 被弾キャラの落下方向を強調する縦の残像

飛び道具:

- キャラごとに色と形を変える。
- 現行の単色円から、コア、外周、尾を持つ弾にする。

### 7.2 ガード / シールド

- 半透明の球体または楕円体シールドを表示。
- シールド耐久が減るほど、サイズ縮小、ひび割れ、色変化を行う。
- シールドブレイク時は破片、白フラッシュ、低音ノイズを出す。

### 7.3 KO

- ブラストゾーン到達時に、キャラ色の光跡を画面外へ伸ばす。
- KO 文字は現行より短く強く表示する。
- 背景を一瞬暗くして、主役を目立たせる。

## 8. 音源仕様

### 8.1 方針

現行の単純な WebAudio 合成音だけではなく、事前制作した音源ファイルを使う。

推奨構成:

- BGM: ループ可能な Ogg / MP3
- SE: 短い Ogg / MP3
- 重要な打撃音のみ、WebAudio で低音やノイズを追加レイヤーしてもよい。

### 8.2 BGM

選択画面:

- 90 から 120 BPM
- 明るいが、ベースとドラムで芯を持たせる。
- 30 秒以上で自然にループ。

バトル:

- 140 から 170 BPM
- ドラム、ベース、リード、コードを含む。
- 60 秒以上で自然にループ。
- 試合開始、残り 1 ストック、勝利時でレイヤー差し替えできると望ましい。

### 8.3 SE

必須 SE:

- menu_move
- menu_select
- countdown
- go
- jump
- double_jump
- land_light
- land_heavy
- jab_hit
- slash_hit
- heavy_hit
- projectile_fire
- projectile_hit
- shield_on
- shield_hit
- shield_break
- ko
- stock_lost
- victory

音作りの要件:

- ヒット音は高音だけにしない。60 から 160 Hz の低域成分を足す。
- スマッシュ、KO、シールドブレイクは専用の低音レイヤーを持つ。
- メニュー音は短く、バトル音と混ざっても邪魔にならない。
- 同じ SE が連打された場合は音量を抑制し、耳に刺さらないようにする。

### 8.4 ミキサー

音量カテゴリ:

- Master
- BGM
- SFX
- UI

仕様:

- ミュートは Master を 0 にする。
- BGM と SFX の音量を個別に設定できる内部構造にする。
- 同時再生数の上限を設ける。
- ヒットストップ中でも音は自然に鳴る。

## 9. UI / HUD 仕様

### 9.1 選択画面

改善点:

- キャラクターカードに 3D モデルの小さなプレビューを表示する。
- 選択済みキャラは輪郭ライトと台座で強調する。
- キャラ性能バーは現行のまま維持しつつ、読みやすい位置に整理する。
- ボタンは Canvas 内描画ではなく、将来は DOM UI へ分離してもよい。

### 9.2 バトル HUD

維持:

- プレイヤー名
- キャラ名
- ダメージ%
- ストック
- シールドゲージ

改善:

- HUD は現行より少し低く、画面端に寄せる。
- 戦闘領域を邪魔しない透明度にする。
- ダメージ%は視認性を最優先し、背景と必ずコントラストを取る。
- キャラアイコンは 3D 顔アイコンまたはレンダー済みポートレートにする。

## 10. 実装設計

### 10.1 推奨ディレクトリ構成

```text
src/
  main.ts
  game/
    loop.ts
    state.ts
    fighter.ts
    physics.ts
    attacks.ts
    cpu.ts
  render/
    renderer.ts
    camera.ts
    stage-view.ts
    fighter-view.ts
    effects.ts
    hud.ts
  audio/
    audio-manager.ts
    sound-map.ts
  assets/
    characters/
    stages/
    audio/
```

### 10.2 既存コードからの移行単位

第1段階:

- 既存 `standalone.html` を Vite + TypeScript に分割する。
- ゲームロジックと Canvas 描画を分離する。
- 見た目は一旦現状維持で動作確認する。

第2段階:

- 描画を Three.js レイヤーへ置き換える。
- `Fighter` の座標、向き、攻撃状態を `fighter-view.ts` に渡す。
- 2D 当たり判定は既存ロジックを維持する。

第3段階:

- キャラクターを低ポリゴン 3D パーツへ置き換える。
- ステージを 3D 化する。
- ヒット、シールド、KO エフェクトを追加する。

第4段階:

- 音源ファイルを導入する。
- BGM と SE のミキサーを実装する。
- イベントごとの音を差し替える。

### 10.3 レンダリング設計

- 1ゲーム座標を 1レンダー単位に近づけ、変換を単純にする。
- X/Y は現行のゲーム座標を使用する。
- Z は以下の用途に限定する。
  - 背景: -100 から -20
  - ステージ: 0
  - キャラクター: 5
  - エフェクト: 10
  - UI 連携用オーバーレイ: DOM または別 Canvas
- キャラクターの向きは `facing` に合わせてモデルの Y 回転または X スケールで表現する。

## 11. アセット仕様

### 11.1 キャラクターモデル

最小要件:

- 各キャラ 1 モデル
- 低ポリゴン
- 主要パーツ分離
- 色替え不要の固有マテリアル
- 1体あたり 500 から 2,500 ポリゴン目安

理想:

- GLB 形式
- Mixamo 風の簡易ボーンまたは独自ボーン
- アニメーションを GLB に同梱
- 1体あたり 300 KB から 1.5 MB 程度

### 11.2 ステージモデル

最小要件:

- メイン足場 GLB
- ソフト足場 GLB
- 背景浮遊島 GLB
- 影受け用透明平面

### 11.3 音源

ファイル形式:

- Ogg を優先
- MP3 フォールバックを用意

長さ目安:

- UI SE: 0.05 から 0.3 秒
- 通常ヒット: 0.1 から 0.4 秒
- スマッシュ/KO: 0.4 から 1.2 秒
- BGM: 30 から 90 秒ループ

## 12. パフォーマンス要件

対象:

- PC Chrome / Edge
- 1280 x 720 以上
- 60 FPS 目標

基準:

- 低スペック PC でも 45 FPS を下回らない。
- 初回ロードは 5 秒以内を目標にする。
- 1フレームあたりの draw call は 100 未満を目安にする。
- 同時パーティクル数は上限を持つ。
- モバイル対応は初回改修では必須にしないが、画面が破綻しないこと。

## 13. 受け入れ基準

### 13.1 見た目

- 戦闘画面を見た瞬間に、現行より奥行きと立体感がある。
- 8キャラが色だけでなくシルエットで識別できる。
- メイン足場とソフト足場の接地可能範囲が明確に見える。
- ヒット、スマッシュ、KO の瞬間がスクリーンショットでも分かる。
- HUD が戦闘キャラを隠さない。

### 13.2 音

- 通常ヒット、スマッシュ、KO、シールドブレイクの音が明確に違う。
- バトル BGM に低音とリズムがある。
- 同時に複数 SE が鳴っても音割れしない。
- ミュートが即時反映される。

### 13.3 ゲーム性

- 1P vs CPU が最後まで遊べる。
- 2P 対戦が最後まで遊べる。
- ストック制、ダメージ%、ノックバックが機能する。
- CPU 難易度切り替えが機能する。

### 13.4 技術

- 音源管理が `AudioManager` に集約されている。
- キャラクターごとの表示差分がデータで管理されている。

## 17. リスクと対策

リスク:

- 3D 化で処理負荷が増える。

対策:

- 低ポリゴン、マテリアル数削減、パーティクル上限、ポストエフェクト最小化を守る。

リスク:

- 3D 表現により当たり判定が分かりにくくなる。

対策:

- ゲーム判定は 2D のまま維持し、攻撃エフェクトを判定範囲に合わせる。

リスク:

- 音源ファイル導入でロードが遅くなる。

対策:

- BGM は遅延ロード、SE は短尺化、形式は Ogg / MP3 に整理する。

リスク:

- キャラモデル制作コストが大きい。

対策:

- 初期は共通リグと共通パーツを使い、頭部、武器、装飾で差別化する。

## 18. 結論

このゲームは、ゲームルールと基本機能はすでに成立している。次に必要なのは、ゲーム性を作り直すことではなく、描画、音源、キャラクター表現を別レイヤーとして強化することである。

最も現実的な改修方針は、既存の 2D 物理と判定を維持しながら、Three.js による 2.5D 低ポリゴン表現へ移行すること。これにより、現行の操作感と開発済みロジックを活かしつつ、ユーザーが求めている「音の重厚さ」「画面の迫力」「キャラクターの立体感」を段階的に実現できる。
